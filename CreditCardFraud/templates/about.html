About ApexGuard

An IoT-ready Machine Learning system for real-time transaction validation.

Project Overview

ApexGuard addresses the critical engineering challenges of modern retail banking environments by bridging high-accuracy offline Machine Learning with ultra-low latency real-time IoT hardware networks. Traditional fraud detection pipelines run asynchronously, alerting users hours or days after a card compromise. ApexGuard is engineered to validate transactions in under 200ms right at the terminal node.

Project Objectives
  • Fraud Deterrence: Block unauthorized charge attempts at the hardware scanning point before transaction authorization is finalized.
  • REST API Interoperability: Expose highly structured endpoints compatible with standard lightweight embedded systems (like ESP32/ESP8266 development boards).
  • Data Ledger & Auditing: Capture and persist transaction logs in a parameterized SQLite storage container for analytics reporting.
  • Dynamic Feature Mapping: Map card scans, location coordinates, transaction times, and amount metrics into clean numerical features suitable for classification.
Core Machine Learning Layer

The prediction service is driven by a serialization pipeline that evaluates features against a trained RandomForestClassifier model. The model is trained on the real Kaggle credit card transaction dataset comprising 284,807 events. The architecture features an isolated preprocessing mapper (`models/iot_feature_mapper.py`) that handles feature normalization, scaling, and PCA projections before feeding the 30-feature vector to the classifier.

Future IoT Development Scope

In future releases, physical ESP32 boards equipped with RFID-RC522 modules and SPI controllers will connect to ApexGuard's endpoints over local Wi-Fi, allowing physical cards or tags to trigger live transaction assessments, outputting instant security alerts on hardware LEDs or LCD displays.

About the Dataset

Context

It is important that credit card companies are able to recognize fraudulent credit card transactions so that customers are not charged for items that they did not purchase.

Content & Imbalance

The dataset contains transactions made by credit cards in September 2013 by European cardholders. It presents transactions that occurred in two days, with 492 frauds out of 284,807 transactions. The dataset is highly unbalanced; the positive class (frauds) accounts for only 0.172% of all transactions.

PCA Transformation

Features V1–V28 are PCA principal components. The only features not transformed are 'Time' (seconds elapsed since the first transaction) and 'Amount' (transaction amount, suitable for cost-sensitive learning). 'Class' takes value 1 for fraud and 0 otherwise.
